## 1. 项目概述

### 1.1 项目背景

本系统为 **B2B2C（Business-to-Business-to-Consumer）** 多商户电商平台。平台运营方提供基础设施和流量入口，商家入驻平台后自主经营，面向终端消费者销售商品。系统同时支持多级分销体系，帮助商家拓展销售渠道。

### 1.2 核心能力矩阵

```
┌─────────────────────────────────────────────────────────────────┐
│                        B2B2C 平台能力矩阵                        │
├──────────┬──────────┬──────────┬──────────┬──────────┬──────────┤
│  多商户   │  商品管理  │  订单履约  │  支付分账  │  分销佣金  │  营销促销  │
│  入驻     │  SPU-SKU  │  状态机   │  统一网关  │  级联树   │  优惠券   │
│  审核     │  多规格   │  拆单发货  │  自动对账  │  冻结结算  │  秒杀活动  │
│  分账     │  价格梯度  │  售后退款  │  虚拟钱包  │  提现审核  │  满减折扣  │
└──────────┴──────────┴──────────┴──────────┴──────────┴──────────┘
```

### 1.3 项目范围

本需求文档覆盖 B2B2C 商城的全量功能需求和非功能性需求，包括：
- PC 商城（`apps/website`，Next.js SSR）、Mobile 移动端（`apps/mobile`，UniApp 3 融合 H5/小程序/App）、商家后台（`apps/seller`）、平台管理后台（`apps/admin`）、供应商后台（`apps/supplier`）五个前端终端
- 商家入驻、商品管理、订单履约、支付结算、营销促销、分销佣金 六大业务域
- 平台运营所需的审核、对账、配置、审计等后台管理功能

## 2. 核心角色定义

### 2.1 角色总览

| 角色 | 说明 | 核心职责 | 权限级别 |
|------|------|----------|----------|
| **平台运营方** | 系统所有者 | 商户入驻审核、平台营销制定、分账规则配置、仲裁处理 | 全平台 |
| **商家主账号（B）** | 入驻商户的店主 | 发布/管理商品、处理订单、管理库存、提现结算、设置子账号权限 | 本店全量 |
| **商家子账号** | 商家员工 | 由主账号分配权限操作特定模块（如仅负责客服/发货） | 本店受限 |
| **消费者（C）** | 终端买家 | 浏览商品、下单支付、收货评价、售后维权 | 仅个人数据 |
| **分销商** | 推广员 | 推广商品赚取佣金，发展下级分销商，提现佣金 | 个人 + 下级数据 |
| **供应商（预留）** | 供货方 | 向平台商家供货，管理供货库存和结算 | 供货相关 |

### 2.2 商家主账号 vs 子账号

| 维度 | 主账号 | 子账号 |
|------|--------|--------|
| 创建方式 | 商家入驻时自动创建 | 主账号在商家后台创建 |
| 权限范围 | 全店功能 | 由主账号分配（RBAC） |
| 数据可见性 | 全店数据 | 按角色限制（如客服仅看订单） |
| 账号数量 | 每商家 1 个 | 每商家最多 50 个 |
| 登录方式 | 手机号 + 密码 | 用户名 + 密码 |

### 2.3 分销商层级规则

- 分销关系为树形结构，最多 **3 级**
- 每一级佣金比例递减（详见 PRD 产品文档）
- 分销商本身不能获得自己订单的佣金（防刷单）
- 分销商申请提现需平台审核

## 3. 业务功能需求

### 3.1 商户生命周期管理

| 编号 | 功能 | 需求描述 | 优先级 |
|------|------|----------|--------|
| M-001 | 商家入驻 | 商家在线提交营业执照、法人身份信息、银行账户等资料 | P0 |
| M-002 | 入驻审核 | 平台运营审核商家资质，通过/拒绝并填写审核意见，通知商家 | P0 |
| M-003 | 店铺管理 | 商家设置店铺名称、Logo、简介、运费模板等 | P0 |
| M-004 | 商家冻结 | 平台可冻结违规商家（可设冻结期限），冻结后商品下架、暂停交易 | P1 |
| M-005 | 商家退店 | 商家主动申请退店，结算完毕后关闭店铺 | P2 |

### 3.2 商品管理

| 编号 | 功能 | 需求描述 | 优先级 |
|------|------|----------|--------|
| P-001 | SPU-SKU 商品模型 | 支持多规格商品（颜色/尺寸/容量等），每 SKU 独立价格/库存/图片 | P0 |
| P-002 | 商品发布 | 商家填写商品信息（标题、描述、图片、规格、价格），提交审核 | P0 |
| P-003 | 商品审核 | 平台审核商品合规性（违禁品、侵权、虚假宣传），通过后上架 | P0 |
| P-004 | 商品搜索 | 关键词搜索、分类筛选、价格排序、属性过滤，结果高亮 | P0 |
| P-005 | 库存管理 | 下单减库存、支付减库存（可配置），库存预警通知商家 | P0 |
| P-006 | 商品上下架 | 商家自主上下架，平台强制下架违规商品 | P1 |

### 3.3 订单与履约

| 编号 | 功能 | 需求描述 | 优先级 |
|------|------|----------|--------|
| O-001 | 购物车 | 支持同商家商品合并、跨商家商品分组展示，实时计算价格 | P0 |
| O-002 | 订单创建 | 从购物车生成订单，填写收货地址、选择优惠券、选择支付方式 | P0 |
| O-003 | 多商家拆单 | 跨商家订单自动拆分为子订单，子订单独立支付/发货/退款 | P0 |
| O-004 | 订单支付 | 支持微信支付、支付宝、余额支付，支付超时自动取消（30分钟） | P0 |
| O-005 | 物流发货 | 商家填写物流单号，系统自动查询物流轨迹 | P0 |
| O-006 | 确认收货 | 用户确认收货或系统超时自动确认（发货后 N 天） | P0 |
| O-007 | 订单评价 | 用户对已确认收货的订单商品进行评分和文字评价 | P1 |

### 3.4 售后退款

| 编号 | 功能 | 需求描述 | 优先级 |
|------|------|----------|--------|
| R-001 | 退款申请 | 用户对未发货/已发货订单申请仅退款或退货退款 | P0 |
| R-002 | 商家处理 | 商家同意/拒绝退款申请，填写处理说明 | P0 |
| R-003 | 退货物流 | 退货退款场景下用户填写退货物流单号 | P0 |
| R-004 | 平台仲裁 | 用户申诉后平台介入处理，强制退款或驳回 | P1 |
| R-005 | 原路退回 | 退款通过原支付渠道退回，金额与原订单一致 | P0 |

### 3.5 支付与财务

| 编号 | 功能 | 需求描述 | 优先级 |
|------|------|----------|--------|
| F-001 | 统一支付网关 | 对接微信支付、支付宝、银联云闪付，统一支付/退款/查询接口 | P0 |
| F-002 | 支付回调处理 | 处理第三方支付回调，验签后更新订单状态，触发分账 | P0 |
| F-003 | 分账结算 | 平台按规则抽佣，剩余资金归属商家，每日自动生成结算单 | P0 |
| F-004 | 每日对账 | 拉取第三方账单与本地支付记录对账，标记差异并生成报告 | P1 |
| F-005 | 虚拟钱包 | 支持用户充值/消费/退款/提现，商家结算资金进入钱包 | P1 |
| F-006 | 商家提现 | 商家申请提现到银行卡/支付宝/微信，平台审核后转账 | P1 |

### 3.6 营销与分销

| 编号 | 功能 | 需求描述 | 优先级 |
|------|------|----------|--------|
| MK-001 | 优惠券管理 | 平台/商家创建满减券、折扣券、无门槛券，设置生效时间和发放量 | P0 |
| MK-002 | 优惠券使用 | 用户在订单中核销优惠券，自动计算优惠价格，防止叠加滥用 | P0 |
| MK-003 | 秒杀活动 | 平台创建限时秒杀活动，设置秒杀价、库存、限购数量 | P1 |
| MK-004 | 满减满折 | 平台/商家设置满减规则（满 N 减 M 或满 N 打折） | P1 |
| MK-005 | 三级分销 | 用户成为分销员后推广商品，下单后上级分销员获得佣金 | P1 |
| MK-006 | 佣金提现 | 分销员可提现可结算佣金，平台审核后转账 | P1 |

### 3.7 平台管理

| 编号 | 功能 | 需求描述 | 优先级 |
|------|------|----------|--------|
| A-001 | 数据仪表盘 | 平台运营后台展示核心经营数据（GMV、订单量、新增用户、新增商家） | P1 |
| A-002 | 商家管理 | 查看商家列表、审核入驻、冻结/解冻商家 | P0 |
| A-003 | 商品审核 | 审核商家发布的商品，通过/驳回 | P0 |

| A-006 | 操作审计 | 记录管理员所有敏感操作日志，支持追溯 | P1 |

## 4. 非功能性需求

### 4.1 性能指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 日活用户（DAU） | 100万+ | 峰值并发按 10% 估算 |
| 峰值 QPS | 50,000+ | 商品列表/详情页为主 |
| 订单峰值 | 10,000 TPS | 秒杀场景下 |
| 系统可用性 | 99.95% | 年度停机时间 < 4.38 小时 |
| 数据一致性 | 强一致性 | 订单、支付、库存必须强一致 |
| 搜索延迟 | P99 < 200ms | Elasticsearch 商品检索 |
| 页面加载 | P99 < 1.5s | 首屏加载时间（CDN + 缓存） |
| API 响应 | P99 < 300ms | 核心接口响应时间 |

### 4.2 安全需求

| 指标 | 目标值 |
|------|--------|
| 传输加密 | 全站 HTTPS，TLS 1.2+ |
| 密码存储 | bcrypt 单向哈希（cost factor ≥ 12） |
| 支付安全 | 第三方回调验签、请求幂等、金额防篡改 |
| 数据加密 | 身份证、银行卡号 AES-256-GCM 加密存储 |
| 接口安全 | SQL 注入防护、XSS 过滤、CSRF Token、限流 |
| 权限控制 | RBAC + 数据权限（merchant_id 隔离） |
| 审计日志 | 敏感操作全量记录，保留 ≥ 180 天 |
| 安全扫描 | 每次发布前 OWASP ZAP 扫描，无高危漏洞 |

### 4.3 兼容性需求

| 维度 | 要求 |
|------|------|
| PC 浏览器 | Chrome 90+, Firefox 88+, Safari 14+, Edge 90+ |
| 移动端 | iOS 13+, Android 8+, 微信 8.0+, 支付宝 10.0+ |
| 小程序 | 微信小程序、支付宝小程序 |
| 屏幕适配 | 响应式设计，支持手机（<768px）、平板（768-1440px）、PC（>1440px） |

### 4.4 扩展性与可维护性

| 需求 | 说明 |
|------|------|
| 水平扩展 | 应用层无状态，支持 K8s HPA 自动扩缩容（3-20 Pod） |
| 数据库扩展 | 当前阶段单库主从，预留分库分表方案（shard key: user_id） |
| 微服务预留 | 核心业务模块按领域划分 Service，未来可按需拆分为独立微服务 |
| 多语言 | 前端支持 i18n，优先中文，预留英文/多语言扩展 |
| API 版本 | URL 路径版本化 `/v1/`，向后兼容 |

### 4.5 数据合规

| 法规 | 要求 |
|------|------|
| 个人信息保护法 | 用户数据收集最小化、提供数据删除/导出接口 |
| 电子商务法 | 订单/支付记录保存 ≥ 3 年 |
| 网络安全法 | 日志留存 ≥ 6 个月，关键数据境内存储 |
| 等保 2.0（三级） | 系统通过三级等保测评 |

## 5. 核心业务流程

### 5.1 商户入驻流程

```
商家注册 -> 填写入驻信息 -> 上传资质 -> 平台审核 -> 审核通过 -> 开通店铺 -> 完善店铺信息 -> 发布商品
                    |
              审核拒绝（返回修改）
```

### 5.2 商品购买流程

```
用户浏览 -> 加入购物车 -> 订单预览 -> 创建订单 -> 支付订单 -> 商家发货 -> 物流运输 -> 确认收货 -> 评价
                                              |
                                        支付超时/取消
```

### 5.3 售后退款流程

```
用户申请退款 -> 商家处理 -> 同意退款 -> 平台退款 -> 资金退回
                    |
              拒绝退款 -> 用户申诉 -> 平台仲裁
```

### 5.4 结算提现流程

```
订单完成 -> 结算周期到达 -> 生成结算单 -> 商家确认 -> 商家申请提现 -> 平台审核 -> 转账到账
```
