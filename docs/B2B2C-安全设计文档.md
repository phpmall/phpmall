## 1. 安全架构概述

### 1.1 安全原则

| 原则 | 说明 |
|------|------|
| **纵深防御** | 不依赖单一安全机制，从网络、应用、数据、业务多层防护 |
| **最小权限** | 每个角色/账号只能访问必要的数据和功能 |
| **零信任** | 内部服务之间也需认证和授权，不默认信任任何请求 |
| **数据分级** | 敏感数据（资金、身份证、银行卡）分级保护，加密存储/传输 |
| **可审计** | 所有敏感操作留痕，可追溯、可分析 |
| **安全左移** | 安全融入开发流程（代码审查、依赖扫描、渗透测试） |

### 1.2 安全威胁模型

```
攻击面：
+--------------------------------------------------+
| 外部攻击者                                        |
|  -> DDoS / CC 攻击 -> 网关层（限流 + WAF）        |
|  -> SQL 注入 / XSS -> 应用层（参数化 + 转义）      |
|  -> 暴力破解 / 撞库 -> 认证层（验证码 + 限流）    |
|  -> 中间人攻击 -> 传输层（HTTPS + 证书固定）      |
|  -> 业务逻辑漏洞 -> 业务层（幂等 + 状态校验）     |
+--------------------------------------------------+
| 内部威胁                                          |
|  -> 员工越权 -> 权限层（RBAC + 数据隔离）         |
|  -> 数据泄露 -> 数据层（加密 + 脱敏 + 审计）       |
|  -> 运维误操作 -> 操作层（双人复核 + 操作日志）    |
+--------------------------------------------------+
```

### 1.3 安全架构图

```
用户 -> CDN -> WAF -> Nginx -> 网关限流 -> 应用层
                                      |
                              +-------+-------+
                              |               |
                         认证服务            业务服务
                        (JWT/黑名单)      (权限校验)
                              |               |
                              +-------+-------+
                                      |
                              +-------+-------+
                              | 数据层安全    |
                              | 加密存储      |
                              | 访问控制      |
                              | 审计日志      |
                              +---------------+
```

## 2. 身份认证安全

### 2.1 密码安全

| 措施 | 实现 |
|------|------|
| 哈希算法 | bcrypt（cost factor = 12），绝对禁止明文/MD5/SHA1 存储 |
| 密码复杂度 | 最少 8 位，包含大小写字母 + 数字 + 特殊字符 |
| 密码历史 | 禁止复用最近 5 次密码 |
| 登录失败锁定 | 连续 5 次失败锁定 30 分钟，或触发图形验证码 |
| 密码传输 | 前端 RSA 公钥加密 -> 后端解密后 bcrypt 哈希（防止中间人嗅探明文） |
| 密码修改 | 必须验证旧密码或短信验证码 |
| 定期更换 | 敏感账号（管理员、财务）强制 90 天更换密码 |

### 2.2 JWT 安全

| 措施 | 实现 |
|------|------|
| 签名算法 | RS256（非对称），私钥签发，公钥验签 |
| 密钥管理 | 私钥存储在 KMS/HSM 或环境变量，禁止代码仓库提交 |
| Token 有效期 | Access Token 2 小时，Refresh Token 30 天（可刷新窗口） |
| Token 刷新 | 旋转 Refresh Token（每次刷新生成新 Token，旧 Token 加入黑名单） |
| Token 黑名单 | Redis 存储注销/刷新的 Token（TTL 设为 Token 剩余有效期） |
| Token 绑定 | 敏感 Token 可绑定设备指纹（User-Agent + IP 哈希） |
| Token 泄露检测 | 同一 Token 在短时间从异地 IP 使用 -> 触发风控告警 + 强制重新登录 |

### 2.3 多端登录安全

| 场景 | 策略 |
|------|------|
| 用户端（Shop/User） | 允许多设备同时登录，最多 5 台设备；可查看/踢出在线设备 |
| 商家端（Seller） | 子账号单设备登录（主账号可强制踢出）；异地登录需二次验证 |
| 管理端（Admin） | 强制单设备登录（新登录踢掉旧）；异地登录短信验证码确认 |
| 敏感操作（支付/提现） | 无论哪端，必须二次验证（支付密码/短信验证码） |

### 2.4 第三方登录安全

| 平台 | 安全措施 |
|------|---------|
| 微信登录 | OAuth 2.0 + code 换 token（禁止前端直接获取 access_token）；校验 openid 与 unionid 绑定关系 |
| 支付宝登录 | 同上，严格校验签名 |
| 短信验证码 | 60 秒间隔，24 小时 10 次上限，5 分钟有效期，5 次错误后失效；防虚拟号段/接码平台（接入手机号风控） |

## 3. 访问控制（RBAC + 数据权限）

### 3.1 RBAC 权限模型

```
用户 -> 角色 -> 权限（permission）
      -> 直接权限（特殊授权）

角色示例：
- 平台：super_admin（超级管理员）、product_admin（商品管理员）、finance_admin（财务管理员）、operator（运营）
- 商家：merchant_owner（店主）、merchant_staff（员工）、merchant_finance（财务）、merchant_customer_service（客服）

权限示例：
- product:create, product:edit, product:delete, product:audit
- order:view, order:ship, order:refund, order:arbitrate
- merchant:view, merchant:audit, merchant:freeze
- finance:view, finance:settlement, finance:withdraw:audit
- system:config, system:admin:create
```

### 3.2 数据权限隔离

| 层级 | 隔离方式 | 说明 |
|------|----------|------|
| 平台管理员 | 全平台数据 | 但角色限制功能权限（如运营不能操作财务） |
| 商家主账号 | 本商家数据 | 通过 `merchant_id` 过滤 |
| 商家子账号 | 本商家 + 角色权限 | 通过 `merchant_id` + 角色权限过滤 |
| 消费者 | 个人数据 | 通过 `user_id` 过滤，禁止访问他人数据 |
| 分销员 | 个人 + 下级数据 | 通过 `distributor_id` 树形过滤 |

**数据权限实现**：
```php
// Laravel 全局 Scope（自动注入）
class MerchantDataScope
{
    public function apply(Builder $builder, Model $model)
    {
        $user = auth()->user();
        if ($user->type === 'merchant_staff') {
            $builder->where('merchant_id', $user->merchant_id);
        }
    }
}

// 水平越权检测（Controller 层）
if ($order->user_id !== auth()->id()) {
    abort(403, '无权访问该订单');
}
```

### 3.3 API 权限控制

- 所有 Admin/Seller API 必须校验权限（`can('permission:action')`）
- 权限变更实时生效（Redis 缓存，TTL 5 分钟，变更时主动清除）
- 未授权访问返回 `403`（不暴露资源是否存在，防止枚举攻击）

## 4. 传输安全

### 4.1 HTTPS 强制

| 环境 | 要求 |
|------|------|
| 生产环境 | 强制 HTTPS，HTTP 301 跳转 HTTPS |
| HSTS | 启用 `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload` |
| TLS 版本 | 最低 TLS 1.2，推荐 TLS 1.3 |
| 证书 | 使用正规 CA 证书，自动续期（Let's Encrypt / 商业证书） |
| 证书固定（可选） | 移动端 App 启用 HPKP 或证书 pinning，防止中间人攻击 |

### 4.2 安全响应头

```http
Strict-Transport-Security: max-age=31536000; includeSubDomains
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
X-XSS-Protection: 1; mode=block
Referrer-Policy: strict-origin-when-cross-origin
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; img-src 'self' https://cdn.phpmall.com data:;
Permissions-Policy: camera=(), microphone=(), geolocation=(self)
```

### 4.3 CORS 策略

```php
// 白名单模式，禁止通配符
'allowed_origins' => [
    'https://admin.phpmall.com',
    'https://www.phpmall.com',
    'https://m.phpmall.com',
    'https://seller.phpmall.com',
]

// 禁止 credentials 为 true 时允许通配符
'supports_credentials' => true,
'allowed_methods' => ['GET', 'POST', 'PUT', 'PATCH', 'DELETE'],
'allowed_headers' => ['Authorization', 'Content-Type', 'X-Request-Id', 'Idempotency-Key'],
'exposed_headers' => ['X-RateLimit-Limit', 'X-RateLimit-Remaining'],
'max_age' => 86400,
```

## 5. 数据安全

### 5.1 敏感数据分级

| 级别 | 数据类型 | 存储要求 | 传输要求 | 展示要求 |
|------|----------|----------|----------|----------|
| P0（极高） | 支付密码、私钥、Token 密钥 | 仅内存/硬件加密，不落盘 | 端到端加密 | 绝不展示 |
| P1（高） | 身份证号、银行卡号、手机号、密码哈希 | AES-256 加密存储 | TLS 传输 | 脱敏展示 |
| P2（中） | 收货地址、真实姓名、订单金额 | 数据库正常存储 | TLS 传输 | 部分脱敏 |
| P3（低） | 昵称、商品信息、公开配置 | 正常存储 | TLS 传输 | 正常展示 |

### 5.2 数据库加密

| 字段 | 加密方式 | 说明 |
|------|----------|------|
| 身份证号 | AES-256-GCM | 数据库层加密，应用层解密 |
| 银行卡号 | AES-256-GCM | 加密存储，展示时脱敏 `**** **** **** 1234` |
| 手机号 | 可选加密 / 脱敏 | 数据库可存明文，但日志/接口必须脱敏 `138****8000` |
| 密码哈希 | bcrypt | 单向哈希，不可解密 |
| 支付密钥 | 环境变量 / KMS | 禁止代码硬编码 |

**PHP 实现示例**：
```php
class SensitiveDataService
{
    private string $key;

    public function encrypt(string $plaintext): string
    {
        $iv = random_bytes(12);
        $ciphertext = openssl_encrypt($plaintext, 'AES-256-GCM', $this->key, OPENSSL_RAW_DATA, $iv, $tag, '', 16);
        return base64_encode($iv . $tag . $ciphertext);
    }

    public function decrypt(string $encrypted): string
    {
        $data = base64_decode($encrypted);
        $iv = substr($data, 0, 12);
        $tag = substr($data, 12, 16);
        $ciphertext = substr($data, 28);
        return openssl_decrypt($ciphertext, 'AES-256-GCM', $this->key, OPENSSL_RAW_DATA, $iv, $tag);
    }

    public function maskPhone(string $phone): string
    {
        return substr($phone, 0, 3) . '****' . substr($phone, -4);
    }

    public function maskIdCard(string $idCard): string
    {
        return substr($idCard, 0, 4) . '**********' . substr($idCard, -4);
    }
}
```

### 5.3 日志脱敏

```php
// 日志中自动脱敏敏感字段
class SensitiveLogProcessor
{
    private array $sensitiveKeys = ['password', 'id_card', 'phone', 'bank_card', 'cvv'];

    public function process(array $record): array
    {
        foreach ($this->sensitiveKeys as $key) {
            if (isset($record['context'][$key])) {
                $record['context'][$key] = '***';
            }
        }
        return $record;
    }
}
```

### 5.4 数据备份安全

- 数据库备份文件加密（AES-256）后上传至对象存储
- 备份保留策略：每日增量保留 30 天，每周全量保留 12 周，每月保留 2 年
- 备份访问权限：仅指定运维角色可访问，操作留审计日志
- 测试环境禁止使用生产数据（如需使用，必须脱敏后导入）

## 6. 支付安全

### 6.1 支付安全核心原则

| 原则 | 实现 |
|------|------|
| **请求签名** | 所有支付请求使用商户私钥签名，服务端验签 |
| **回调验签** | 第三方支付回调必须验签，防止伪造回调 |
| **防重放攻击** | 请求包含时间戳 + 随机数，服务端校验时间窗口（±5分钟）和 nonce 唯一性 |
| **金额防篡改** | 订单金额服务端计算，前端只传商品/优惠券 ID，金额由后端重新计算比对 |
| **幂等控制** | 支付请求携带 Idempotency-Key，防止重复支付 |
| **状态机校验** | 支付前校验订单状态必须为「待支付」，防止已支付/已取消订单重复支付 |
| **退款安全** | 退款金额必须 <= 实付金额，退款单号唯一，原路退回 |

### 6.2 支付流程安全

```
用户下单 -> 服务端生成订单（计算金额） -> 返回订单信息（不含金额给前端）
用户确认支付 -> 前端请求支付接口（仅传 order_id + pay_method）
服务端校验订单 -> 重新计算金额 -> 调用支付渠道（服务端签名）
支付渠道返回 -> 服务端验签 -> 幂等校验 -> 更新订单状态 -> 异步分账
第三方支付回调 -> 验签 -> 幂等校验 -> 更新订单 -> 发送通知
```

### 6.3 支付密钥管理

| 密钥 | 存储方式 | 说明 |
|------|----------|------|
| 微信支付 APIv3 密钥 | 环境变量 + KMS | 定期轮换（90 天） |
| 支付宝私钥 | 环境变量 + KMS | 定期轮换 |
| 证书（微信/支付宝） | 文件系统 + 只读权限 | 有效期监控，到期前告警 |
| 分账密钥 | 独立环境变量 | 与支付密钥隔离 |

### 6.4 支付风控

| 规则 | 触发动作 |
|------|----------|
| 同一用户 1 分钟内 > 3 笔支付 | 弹窗确认 + 短信验证 |
| 同一用户单日支付金额 > 5 万 | 触发大额交易验证（人脸识别/身份证） |
| 异地支付（IP 突变） | 弹窗确认 + 短信验证 |
| 高风险设备（模拟器、越狱设备） | 拒绝支付或强制二次验证 |
| 支付渠道返回异常率 > 50% | 熔断该渠道，切换备用渠道 |

## 7. 接口安全

### 7.1 SQL 注入防护

| 措施 | 实现 |
|------|------|
| 参数化查询 | 所有数据库查询使用 PDO prepared statements / Eloquent ORM |
| 禁止字符串拼接 | 绝对禁止 `where('id = ' . $id)` 形式 |
| 动态排序白名单 | 列表排序字段必须在白名单内，禁止用户传入任意字段名 |
| 输入过滤 | 特殊字符（单引号、分号、注释符）转义 |
| 工具扫描 | SQLMap 定期扫描，SonarQube 代码规则检测 |

### 7.2 XSS 防护

| 措施 | 实现 |
|------|------|
| 输出编码 | 所有用户输入输出到 HTML 时进行 HTML entity 编码（`htmlspecialchars`） |
| CSP 策略 | 配置 `Content-Security-Policy` 限制脚本来源 |
| 富文本过滤 | 商品详情等富文本使用 DOMPurify 或类似库过滤危险标签（`<script>`, `<iframe>`, `onerror` 等） |
| Cookie 安全 | `HttpOnly; Secure; SameSite=Strict` |
| 存储型 XSS 检测 | 用户输入内容（评论、商品描述）入库前过滤危险标签 |

### 7.3 CSRF 防护

| 端 | 防护方式 |
|----|---------|
| 管理后台（传统 Web） | Laravel CSRF Token + SameSite Cookie |
| 前后端分离（SPA） | JWT 认证（无 Cookie 会话），天然免疫 CSRF |
| 小程序/App | Token 认证，无 Cookie |

### 7.4 接口限流与防刷

| 场景 | 策略 | 实现 |
|------|------|------|
| 全局 IP 限流 | 匿名 60 次/分钟 | Redis + 滑动窗口 |
| 用户 Token 限流 | 300 次/分钟 | Redis + 滑动窗口 |
| 短信验证码 | 60 秒/次，24h 10 次 | Redis + 计数器 |
| 登录尝试 | 5 次/分钟，失败 5 次锁定 30 分钟 | Redis + 计数器 |
| 支付接口 | 10 次/分钟 | Redis + 令牌桶 |
| 注册接口 | 同一 IP 10 次/小时 | Redis + 计数器 |
| 暴力枚举 | 连续 404/403 超过阈值封 IP | WAF + 防火墙 |

### 7.5 敏感接口二次验证

| 接口 | 二次验证方式 |
|------|-------------|
| 支付订单 | 支付密码 / 短信验证码 |
| 钱包提现 | 短信验证码 + 支付密码 |
| 修改绑定手机 | 原手机验证码 + 新手机验证码 |
| 修改密码 | 原密码 或 短信验证码 |
| 实名认证 | 人脸识别 + 身份证 OCR |
| 商家结算确认 | 短信验证码 |
| 平台管理员敏感操作 | 短信验证码 + 操作审计 |

## 8. 文件上传安全

### 8.1 文件安全校验

| 检查项 | 实现 |
|--------|------|
| 扩展名白名单 | 仅允许 `jpg`, `jpeg`, `png`, `gif`, `webp`, `pdf` |
| MIME 类型校验 | 通过 `finfo_file` 获取真实 MIME，与扩展名交叉验证 |
| 文件头魔数 | 校验文件头（如 `FFD8FF` 为 JPG） |
| 文件大小 | 严格限制，超过拒绝 |
| 图片重编码 | 上传图片后使用 GD/Imagick 重新编码，去除 EXIF 信息（防止 GPS 泄露） |
| 病毒扫描 | 使用 ClamAV 或云扫描引擎扫描上传文件 |
| 存储隔离 | 用户上传文件存储在独立目录（非 Web 根目录），通过 CDN 代理访问 |

### 8.2 防止上传漏洞

| 漏洞 | 防护 |
|------|------|
| 上传可执行脚本 | 禁止 `.php`, `.jsp`, `.asp`, `.sh` 等；文件名重命名（UUID） |
| 路径遍历 | 文件名去除 `../`，使用白名单目录 |
| 二次渲染绕过 | 所有图片重新编码后存储 |
| SVG 脚本 | 禁止使用 SVG 上传（或严格过滤 `<script>`） |
| 大文件 DoS | 限制大小 + 超时控制 |

## 9. 审计与日志安全

### 9.1 审计日志范围

| 操作类型 | 必须记录 |
|----------|---------|
| 认证 | 登录成功/失败、登出、Token 刷新、密码修改 |
| 用户数据 | 手机号/邮箱绑定修改、实名认证、地址增删改、账号注销 |
| 商家数据 | 入驻提交、审核通过/拒绝、冻结/解冻、分账配置修改 |
| 商品数据 | 商品创建/修改/删除/上下架、价格/库存变更、审核操作 |
| 订单数据 | 订单创建、支付、发货、确认收货、取消、退款申请/处理 |
| 财务数据 | 支付记录、退款记录、分账记录、结算确认、提现申请/审核/转账 |
| 权限数据 | 角色创建/修改、权限分配、管理员创建/禁用/密码重置 |
| 配置数据 | 系统配置变更、支付方式开关、结算规则变更 |
| 安全事件 | 异地登录、异常支付、权限越权、敏感接口访问 |

### 9.3 日志安全

| 措施 | 实现 |
|------|------|
| 日志传输 | 日志通过 TLS 加密传输到日志中心（ELK / Loki） |
| 日志存储 | 日志保留 180 天，敏感操作日志保留 2 年 |

## 10. 第三方集成安全

### 10.1 微信支付安全

| 措施 | 实现 |
|------|------|
| APIv3 | 使用微信支付 APIv3，TLS 1.2+，RSA 签名 |
| 回调验签 | 必须使用微信支付平台证书验签回调 |
| 回调 IP 白名单 | 仅允许微信官方 IP 段访问支付回调接口 |
| 通知幂等 | 同一回调通知号（`id`）只处理一次 |
| 敏感信息 | 不存储用户支付密码、CVV、磁条数据 |

### 10.2 支付宝安全

| 措施 | 实现 |
|------|------|
| RSA2 签名 | 使用 RSA2（SHA256WithRSA）签名 |
| 回调验签 | 使用支付宝公钥验签回调 |
| 异步通知 | 校验通知 ID 幂等，处理成功后返回 `success` |
| 沙箱测试 | 所有支付流程必须在沙箱环境完整测试后上线 |

### 10.3 物流/短信/OSS 安全

| 第三方 | 安全措施 |
|--------|---------|
| 物流查询 | 使用 HTTPS 接口，缓存物流数据，防止 API 滥用 |
| 短信服务 | 模板预审核，禁止动态拼接短信内容（防止钓鱼）；限制发送频率 |
| 对象存储（OSS） | Bucket 私有读写，通过预签名 URL 临时授权；CDN 回源鉴权 |
| 地图服务 | 密钥绑定域名/IP，防止盗用；用户地址数据不发送到地图服务商 |

## 11. 安全事件响应

### 11.1 安全事件分级

| 级别 | 定义 | 响应时间 | 处理时间 | 通报范围 |
|------|------|----------|----------|----------|
| P0 | 数据泄露、资金盗取、系统被入侵、大规模用户影响 | 15 分钟 | 4 小时 | CTO + 安全团队 + 法务 |
| P1 | 支付漏洞、权限绕过、核心接口可被攻击 | 1 小时 | 24 小时 | 技术负责人 + 安全团队 |
| P2 | XSS/CSRF 可利用、信息泄露（非敏感）、配置错误 | 4 小时 | 3 天 | 安全团队 + 开发团队 |
| P3 | 低危漏洞、扫描告警、策略优化建议 | 1 天 | 1 周 | 安全团队 |

### 11.2 应急响应流程

```
发现异常 -> 确认事件级别 -> 成立响应小组 -> 止血（隔离/下线/回滚）
    -> 根因分析 -> 修复 -> 验证 -> 复盘 -> 改进 -> 归档
```

**止血措施**：
- 数据泄露：立即切断泄露源，冻结相关账号，启动数据影响评估
- 资金风险：暂停支付/提现接口，冻结可疑资金，联系支付渠道冻结
- 系统入侵：隔离被入侵服务器，保留现场日志，启动 forensic 分析
- 漏洞利用：临时 WAF 规则拦截，紧急修复后上线

### 11.3 安全监控告警

| 告警规则 | 告警级别 | 响应 |
|----------|---------|------|
| 敏感接口 403 异常突增（> 100 次/分钟） | 中 | 排查越权扫描 |
| 同一账号异地登录（城市距离 > 500km） | 高 | 通知用户 + 强制二次验证 |
| 支付失败率突增（> 30%） | 高 | 排查支付渠道/攻击 |
| 数据库慢查询突增 | 中 | 排查 SQL 注入/攻击 |
| 新漏洞 POC 出现（CVE） | 高 | 评估影响，紧急修复 |
| 代码依赖出现高危漏洞 | 中 | 升级依赖版本 |
| 证书即将过期（< 30 天） | 低 | 自动续期或人工更换 |

## 12. 安全合规要求

### 12.1 合规清单

| 合规 | 要求 | 说明 |
|------|------|------|
| 等保 2.0（三级） | 网络安全等级保护 | 系统需通过三级等保测评 |
| 个人信息保护法 | 用户数据收集最小化、知情同意、可删除 | 隐私政策、用户协议、数据删除接口 |
| 数据安全法 | 数据分类分级、加密存储、跨境传输限制 | 敏感数据不出境，加密存储 |
| 电子商务法 | 商家资质审核、交易记录保存、消费者权益保护 | 订单/支付记录保存 >= 3 年 |
| 网络安全法 | 日志留存 >= 6 个月、关键数据境内存储 | 日志、用户数据境内存储 |
| PCI DSS（可选） | 支付卡行业数据安全标准 | 如涉及银行卡直接支付，需满足 PCI DSS |

### 12.2 合规措施

| 措施 | 实现 |
|------|------|
| 隐私政策 | 明确告知用户数据收集范围、用途、存储期限、第三方共享 |
| 用户协议 | 明确用户权利（账号注销、数据导出、投诉渠道） |
| 数据最小化 | 仅收集业务必需信息（不过度收集通讯录、位置等） |
| 数据删除 | 提供账号注销功能，注销后 30 天内物理删除数据（法律法规要求的留存除外） |
| 数据导出 | 用户可导出个人数据（订单、地址、钱包流水） |
| 未成年人保护 | 实名认证识别未成年人，限制消费金额，禁止夜间交易 |
| 商家资质审核 | 营业执照、法人身份证、银行账户核验，确保商家真实合法 |
