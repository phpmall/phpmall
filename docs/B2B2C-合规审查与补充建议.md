# B2B2C 合规审查与补充建议

> 审查范围：`docs/` 全部文档、`app/Modules/` 领域模块。
> 审查视角：财务合规 + 法务合规。
> 说明：本报告为技术/产品合规 gap 分析，不构成正式法律或审计意见。

---

## 一、总体结论

当前文档与领域模块在**交易闭环、支付安全、对账、审计日志、商家入驻基础审核、订单售后功能、数据安全技术防护**等方面已有较好覆盖，基本满足 MVP/V1.0 业务需求。

但从**财务合规与金融监管、法务合规与平台责任**角度看，存在以下明显短板：

1. **资金合规架构（二清/存管）** 未在架构层面闭环；
2. **税务发票** 几乎空白；
3. **反洗钱/可疑交易监控** 仅停留在简单风控规则；
4. **分销佣金/商家提现的税务代扣代缴** 未涉及；
5. **用户协议/隐私政策/PIPL 数据主体权利** 仅原则性提及；
6. **电子合同/交易存证** 未明确；
7. **商品合规准入、知识产权投诉、广告法审查** 缺少专门流程；
8. **消费者权益保护规则、投诉调解、评价治理** 需进一步细化。

建议在 V1.0 → V1.5 阶段优先补齐高优先级合规模块，否则系统上线后可能面临较大的合规整改成本。

## 二、已覆盖的合规能力

| 合规领域 | 已覆盖内容 | 所在模块/文档 |
|---|---|---|
| 统一支付网关 & 第三方通道 | 微信支付、支付宝、银联接入，回调验签、幂等、金额防篡改、限流熔断 | `Payment`、技术方案 §6.1 |
| 退款原路退回 | 退款金额 ≤ 实付金额，唯一退款单号 | `Refund`、安全设计 §6.1 |
| 平台-商户分账 | 订单级分账，平台服务费 + 商家货款 | `Payment`、技术方案 §6.2 |
| T+N 结算 & 商家提现审核 | 结算周期、结算单、提现到卡/支付宝/微信、平台审核 | `Finance` / `Merchant`、PRD §8.6 |
| 每日对账 | 按渠道拉账单，差异标记 | `Finance`、技术方案 §6.3 |
| 审计日志 / 操作留痕 | 财务相关操作留痕，日志保留策略 | `System` / `AuditLog`、安全设计 §9 |
| 敏感数据加密 & 脱敏 | 身份证、银行卡号 AES-256-GCM，日志脱敏 | 安全设计 §5 |
| 商家入驻资质审核 | 营业执照、法人身份证、银行账户核验 | `Merchant`、安全设计 §12.2 |
| 商品审核与违规处置 | 商品审核、强制下架、商家冻结 | `Product` / `PlatformOperation` |
| 订单/售后/消费者权益（功能） | 订单状态机、7 天内售后、平台仲裁、评价 | `Order` / `Refund` |
| 三级分销合规（部分） | 分销层级 ≤3 级、禁止自购佣金、佣金冻结结算 | `Distribution` |
| 网络安全与等保 2.0 | HTTPS/TLS、WAF、限流、防爬、SQL/XSS/CSRF 防护 | 安全设计 §4/§7 |
| 身份认证与访问控制 | bcrypt、JWT RS256、RBAC + 数据权限 | `Auth`、安全设计 §2/§3 |

## 三、财务合规遗漏与补充建议

### 高优先级

| 遗漏点 | 风险说明 | 建议补充模块 | 相关法规/要求 |
|---|---|---|---|
| **二清（二次清算）合规架构** | 平台收款后再分账/结算给商户，若未取得支付牌照，属于“大商户二清”违规 | `Escrow` 资金存管域 | 人民银行《非金融机构支付服务管理办法》 |
| **税务发票管理（电子发票）** | 平台服务费发票、商家销售发票、佣金个税发票缺失 | `Invoice` 发票税务域 | 《发票管理办法》、数电发票政策 |
| **反洗钱（AML）/可疑交易监控** | 缺少可疑交易识别、客户风险等级、STR、名单筛查 | `Compliance` 合规风控域 | 《反洗钱法》、人民银行支付机构反洗钱规定 |
| **资金监管 / 备付金存管** | 虚拟钱包充值、商家待结算资金沉淀需专户存管 | `Escrow` 资金存管域 | 《支付机构客户备付金集中存管办法》 |
| **分销佣金个税代扣代缴** | 三级分销佣金提现到个人，平台需履行代扣代缴义务 | `Invoice` / `Distribution` | 《个人所得税法》 |

### 中优先级

| 遗漏点 | 建议补充 |
|---|---|
| 电子回单与支付凭证 | `Invoice` / `Finance` 增加电子回单生成与签章 |
| 营销优惠财税分摊 | `Marketing` / `Order` 明确平台券/商家券承担方与开票分摊 |
| 退款与已开发票联动 | `Invoice` / `Refund` 退款触发红冲发票流程 |
| 商家提现账户一致性校验 | `Merchant` / `Finance` 提现账户变更需二次审核 |
| 平台抽佣价税分离 | `Finance` 拆分含税/不含税金额 |

### 低优先级

| 遗漏点 | 说明 |
|---|---|
| 完整财务报表体系 | 资产负债表、利润表、现金流量表、科目余额表 |
| 会计科目与账务分录 | 自动生成记账分录 |
| 积分/虚拟资产监管 | 若积分可抵现需按预付价值监管 |
| 跨境支付/外汇合规 | 为 V2.0 生态化预留 |

## 四、法务合规遗漏与补充建议

### 高优先级

| 遗漏点 | 风险说明 | 建议补充模块 | 相关法规/要求 |
|---|---|---|---|
| **用户协议 & 隐私政策管理** | 仅原则性提及，缺少条款结构、同意记录、用户权利接口 | `DataPrivacy` 数据合规与隐私域 | 《个人信息保护法》《网络安全法》 |
| **特殊商品/商家合规准入** | 缺少特殊行业许可证、品牌授权、禁限售目录 | `ProductCompliance` / `Merchant` | 《电子商务法》《食品安全法》《产品质量法》 |
| **电子合同与交易存证** | 订单/协议生效时点、电子签名、可信存证不明确 | `Contract` 电子合同与法务域 | 《电子商务法》《电子签名法》 |
| **支付/钱包/分账 AML 与税务** | 虚拟钱包、佣金提现涉及资金合规与发票 | `Compliance` / `Invoice` | 《反洗钱法》《发票管理办法》 |
| **评价治理与反不正当竞争** | 禁止删除差评、好评返现、刷单炒信 | `ConsumerProtection` / `ProductCompliance` | 《电子商务法》《反不正当竞争法》 |

### 中优先级

| 遗漏点 | 建议补充 |
|---|---|
| 消费者权益保护规则细化 | 7 天无理由退货、三包、假货赔付、运费承担 |
| 知识产权侵权投诉处理 | 通知-删除流程、反通知申诉、侵权处罚公示 |
| 广告/营销合规 | 极限用语审查、价格欺诈防范、医疗/保健广告限制 |
| 争议解决与投诉渠道 | 在线纠纷解决（ODR）、12315 对接、管辖条款 |
| 平台责任与免责声明 | 平台责任边界、审查义务范围、不可抗力免责 |
| 未成年人保护 | 年龄验证、大额消费拦截、夜间交易限制、家长监护 |

### 低优先级

| 遗漏点 | 说明 |
|---|---|
| 数据跨境传输与 GDPR | 仅出境场景适用，架构预留 |
| 直播电商合规 | 主播实名、内容审核、打赏税务 |
| 反垄断与算法透明度 | 搜索排序透明度、反大数据杀熟 |

## 五、新增领域模块清单

基于上述审查，已在 `app/Modules/` 新增以下 8 个合规领域模块：

| 模块 | 类型 | 职责 | 对应合规领域 |
|---|---|---|---|
| `Escrow` | 核心域 | 资金存管、用户/商户资金隔离、保证金、押金 | 二清合规、备付金存管 |
| `Invoice` | 支撑域 | 电子发票、红冲/作废、代扣代缴、纳税申报 | 税务发票、个税代扣 |
| `Compliance` | 核心域 | KYC、AML、反欺诈、制裁名单筛查、风险等级 | 反洗钱、实名认证 |
| `DataPrivacy` | 核心域 | 隐私政策、同意管理、数据主体权利、未成年人保护 | PIPL/GDPR |
| `Contract` | 支撑域 | 电子合同、合同模板、电子签名、可信存证 | 电子商务法、电子签名法 |
| `ConsumerProtection` | 支撑域 | 投诉受理、平台调解、先行赔付、7 天无理由规则 | 消费者权益保护法 |
| `ProductCompliance` | 核心域 | 违禁品、侵权、广告法、敏感内容、资质准入 | 电子商务法、广告法、知识产权法 |
| `AuditLog` | 通用域 | 跨模块审计事件、WORM 存储、合规报表 | 等保 2.0、审计留痕 |

## 六、现有模块合规职责更新建议

| 现有模块 | 建议补充的合规职责 |
|---|---|
| `Merchant` | 特殊行业许可证审核、资质到期复检、提现账户变更复核、保证金缴纳/扣划 |
| `Product` | 商品合规审核接入 `ProductCompliance`、禁限售目录、侵权投诉联动 |
| `Order` | 交易记录归档 ≥3 年、电子合同签署时点、消费者权益规则触发 |
| `Payment` | 大额/可疑交易上报 `Compliance`、支付回单、PCI DSS 合规 |
| `Finance` | 资金隔离对接 `Escrow`、价税分离、电子回单、合规报表 |
| `Refund` | 退款触发发票红冲、原路退回留痕、售后证据保全 |
| `Distribution` | 分销佣金个税代扣、反传销合规、推广费结算凭证 |
| `Marketing` | 优惠财税分摊、广告法审查、促销规则显著披露 |
| `Content` | 隐私政策/用户协议版本管理由 `DataPrivacy` 主导，Content 负责静态展示 |
| `System` | 基础操作日志由 `AuditLog` 统一升级，支持 WORM 与合规报表 |
| `User` | 扩展为会员域，纳入实名认证、隐私设置、数据导出/注销 |

## 七、下一步行动建议

1. **高优先级（V1.0 前必须明确架构）**
   - 确定资金存管方案：持牌支付机构“平台二级商户”模式 vs 银行存管账户模式；
   - 明确发票开具责任主体与对接税局/第三方电子发票服务商；
   - 设计 KYC/AML 最小可行流程：实名认证、大额交易监测、可疑交易报告；
   - 制定隐私政策/用户协议条款结构与用户同意记录方案。

2. **中优先级（V1.0 → V1.5）**
   - 补齐电子合同签署与存证流程；
   - 建立商品合规准入与知识产权投诉处理流程；
   - 完善消费者权益规则引擎（7 天无理由、三包、赔付）；
   - 实现评价治理与反刷单机制。

3. **低优先级（V1.5 之后）**
   - 财务核算与三大报表；
   - 跨境支付与 GDPR 兼容；
   - 直播电商合规、算法透明度。
